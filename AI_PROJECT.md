## Reusable Notes

- Symptom: GitLab CI `release_version` job fails with `Committer identity unknown` when creating an annotated tag. Cause: the runner container does not have `git user.name` and `git user.email` configured by default. Fix: set both values in the release job `before_script` before running `git tag -a` or other write operations.
- Symptom: GitLab CI `release_version` job fails with `Host key verification failed` while pushing a tag back to GitLab over SSH. Cause: `known_hosts` only trusts the deploy server, but `git push` targets `CI_SERVER_HOST`, so the runner does not trust the GitLab SSH host key. Fix: add `ssh-keyscan "${CI_SERVER_HOST}" >> ~/.ssh/known_hosts` before pushing, and ensure the SSH key used by CI is also authorized to write to the GitLab repository.
- Symptom: normal pushes to `main` unintentionally trigger version release logic in CI, mixing routine deployment with external release publishing. Cause: release responsibilities were implemented inside the GitLab pipeline instead of the dedicated local release script. Fix: keep CI limited to build and deploy for `main`, and let `build.sh` alone handle version bump, commit, tag creation, and release publication.
- Symptom: a manual code commit followed by a separate release commit causes duplicate `main` pushes and therefore duplicate deployments. Cause: release automation only committed `package.json`, forcing users to push code changes separately before running `build.sh`. Fix: `build.sh` should stage all current local changes together with the version bump and create a single release commit before pushing `main`.
- Symptom: the frontend shows a blank screen after production build, with runtime errors around `crypto-js` internals such as `WordArray` being undefined. Cause: Vite production output was processed by both an aggressive Terser pass and a second JavaScript obfuscation step, which together can break runtime-sensitive libraries. Fix: keep only one obfuscation stage for frontend assets and avoid stacking aggressive minification and obfuscation on the same bundle.
- Symptom: the first-install flow shows takeover-related hints or accepts reuse of old environment data, which conflicts with the project requirement of clean installs only. Cause: the setup page and backend still preserved an old-environment takeover path. Fix: keep setup strictly as pure first install, remove takeover fields and messages from the UI, and reject initialization when the target database already contains system tables.
- Symptom: the first-install page looks awkward when it reuses the narrow login-card layout for multiple setup fields. Cause: the setup form has more inputs and needs clearer grouping than the login form, but both screens shared the same compact card structure. Fix: use a wider dedicated setup card and split the form into clear sections for database connection and administrator information.
- Symptom: first-install page spacing keeps drifting to the right even after repeated CSS tweaks. Cause: the page reuses generic classes such as `search-input`, `auth-input`, and shared grid helpers, so inherited width and layout rules override setup-specific spacing. Fix: give the setup page its own dedicated layout and input classes, and avoid mixing it with shared login/search form styles.
- Symptom: admin login or setup accepts usernames with symbols, causing inconsistent behavior between screens and stored admin accounts. Cause: administrator username format was not constrained consistently in login, setup, and admin management flows. Fix: restrict administrator usernames to English letters and digits in both frontend input sanitization and backend validation.
- Symptom: node configuration allowed direct entry of arbitrary `ssh_private_key_path`, which could expose local server paths and make SSH key loading depend on uncontrolled filesystem locations. Cause: the node form trusted raw private-key paths instead of using a managed upload flow. Fix: require SSH key authentication to use uploaded private/public key files stored under a controlled `storage/ssh-keys` directory, return only safe file labels to the frontend, and redact internal paths from user-facing error messages.
- Symptom: low-privilege viewer accounts could access remote logs or sensitive system settings pages, exposing operational details and allowing unintended configuration changes. Cause: role permission maps granted `logs.view` and `appearance.manage` too broadly in both backend and frontend permission definitions. Fix: keep viewer limited to basic read-only panels, remove high-sensitivity permissions from viewer, and reserve system-setting changes for trusted roles only.
- Symptom: setup status and unexpected backend errors could leak database or filesystem details to the browser. Cause: setup checks and global exception handling forwarded raw exception messages or stack traces to clients. Fix: return only generic user-facing failure messages, redact internal paths, and keep detailed diagnostics out of API responses.
- Symptom: deployment documents accidentally contained a real production site path/domain, which would expose internal environment details once the repository is public. Cause: deployment examples were written directly from the live environment instead of using placeholders. Fix: keep all repository docs and examples on generic placeholder domains/paths such as `panel.example.com`, and never commit real website identifiers.
- Symptom: running a frontend production build leaves old hashed assets in `public/assets`, which can bloat releases or expose stale code. Cause: Vite writes new hashed filenames while previous build outputs remain on disk. Fix: after a successful build, keep only assets referenced by `public/index.html` and remove unreferenced hashed artifacts.
- Symptom: adding `echarts` with a full-package import makes the frontend production bundle grow sharply and bloats the obfuscated release artifact. Cause: `import * as echarts from 'echarts'` pulls in the full charting runtime instead of only the required modules. Fix: register only the needed ECharts pieces from `echarts/core`, such as the line chart, grid, legend, tooltip, and canvas renderer.
- Symptom: the user management page can become visually broken with misaligned columns and squeezed action buttons after layout tweaks. Cause: a list-style grid makes the template column count and CSS track definition easy to desynchronize, especially across responsive breakpoints. Fix: keep the user page in list form, but treat the row template and every breakpoint track definition as one unit, and switch to labeled stacked cells on narrow screens instead of ad-hoc column collapse.
- Symptom: opening the dashboard or large list pages causes unnecessary payload growth and slower rendering as nodes, users, logs, or traffic data accumulates. Cause: list endpoints and `panel/state` aggregate responses return full collections, while the frontend slices them locally. Fix: keep `panel/state` as a lightweight summary payload, move multi-record views to server-side pagination, and request realtime stats only for the records shown on the current page.
- Symptom: the desktop admin backend grows taller than the browser viewport and feels messy once cards or lists become longer. Cause: the shell and page containers rely on document height, so content keeps expanding instead of being constrained by the viewport. Fix: on desktop, keep the backend shell at `100vh`, make the sidebar and main content area the explicit scroll containers, and only fall back to natural page height on stacked tablet/mobile breakpoints.
- Symptom: enabling near-realtime Agent heartbeats makes the node traffic history table grow far too fast and pollutes trend sampling. Cause: every heartbeat can fire every few seconds or faster, but historical charts only need periodic snapshots. Fix: keep Agent heartbeats limited to updating the latest cached metrics in `node_agents`, and write `node_traffic_history` only from explicit snapshot/sync flows.
- Symptom: after deploying new backend code, operators still have to manually run SQL migration files before the panel or CLI can use new fields and tables. Cause: the project originally imported only `schema.sql` during first install and had no automatic upgrade step for existing databases. Fix: run a shared migration service automatically in Web and CLI entrypoints, keep a `schema_migrations` table, and mark bundled migrations as applied after fresh installs.
- Symptom: generated project docs can accidentally expose the maintainer's local workstation path, such as absolute `file:///Users/...` links. Cause: documentation was written with IDE-local absolute links instead of repository-relative references. Fix: keep repo docs on relative paths like `docs/LLM_HANDOFF.md`, and never commit local filesystem URLs or desktop paths.
- Symptom: mock mode crashes on startup with `Cannot access 'runtimeStatuses' before initialization` inside `mockPanelData.ts`. Cause: top-level mock node creation calls `createNode()` before `runtimeStatuses` is initialized, triggering the ES module temporal dead zone. Fix: declare shared runtime status maps before any top-level factory calls that read them, or pass the needed status into the factory explicitly.
- Symptom: changing panel branding only in the admin settings form does not update the browser tab for login or low-permission users. Cause: title/icon settings were only fetched through the privileged `system-settings` endpoint after entering the backend. Fix: expose a minimal public app-settings endpoint for safe branding fields like `site_title` and `site_icon_url`, and apply them during frontend bootstrap before auth-dependent flows.
- Symptom: the sidebar brand area looks too busy when it shows both the English brand line and the extra Chinese `控制面板` heading, especially after the version badge is moved directly below it. Cause: duplicate title layers in the top-left brand block compete for space in the fixed-height sidebar. Fix: keep only the single brand line in the sidebar brand card, enlarge that line appropriately, and merge adjacent appearance settings cards when they belong to one visual group.
- Symptom: creating a node with agent mode fails with `ssh2_connect(): Error starting up SSH connection(-43): Failed getting banner`, while SSH mode with the same config works fine. Cause: `RemoteExecutor::run()` creates a new SSH connection on every call but never explicitly disconnects. The install flow for agent mode makes 5 sequential SSH calls, accumulating 4 unreleased connections before the 5th one fails due to SSH server or PHP ssh2 extension connection limits. Fix: add `ssh2_disconnect()` in a `try-finally` block in `run()`, add a `checkPortReachable()` pre-check with a `fsockopen` probe for better error diagnostics, and translate ssh2 warnings into safe user-facing messages.
- Symptom: agent deployment fails with `Argument exceeds the allowed length of 2097152 bytes`. Cause: the bootstrap command embeds multi-megabyte Agent binaries as base64 in a single `ssh2_exec()` command, exceeding the SSH extension argument limit. Fix: detect the remote CPU architecture first, expose the matching Agent binary through a short-lived signed download URL, and let SSH run only a small `curl`/`wget` bootstrap command.
- Symptom: creating a new node can leave a saved node record behind when the automatic SSH install fails. Cause: the old frontend flow saved the node first and then called the install endpoint separately, so a failed install left the already-created record visible. Fix: use a backend create-and-install endpoint for new-node flows that creates the temporary DB record needed for install tokens, deletes it if install throws or returns a non-zero exit code, and returns the node only after install succeeds.
- Symptom: realtime log audit mode refreshes data but leaves the viewer at an old scroll position. Cause: auto refresh updates `logsText` and `auditLogs` without adjusting their scroll containers after Vue renders the new content. Fix: keep refs for the node log terminal and audit list, then on realtime refresh use `nextTick` to scroll node logs to the bottom and audit records to the top where newest entries appear.
- Symptom: realtime log polling can overload SSH-managed nodes when the log audit page refreshes frequently. Cause: the realtime timer did not distinguish Agent-managed nodes from SSH-managed nodes. Fix: only start high-frequency realtime polling for selected nodes that have Agent mode enabled and an Agent record; keep manual refresh available for SSH or undeployed-Agent nodes.
- Symptom: Agent logs repeatedly show `cannot unmarshal array into Go struct field task.data.task.payload of type map[string]interface {}` and some Agent tasks stay `running`. Cause: PHP encodes an empty array payload as JSON `[]`, but the Go Agent expects task payloads to be objects. Fix: encode empty Agent task payloads as `{}` on the panel side and make the Agent tolerate legacy `[]` payloads as empty objects.
- Symptom: node log viewing depends on `journalctl`, mixing application logs with system journal behavior. Cause: Agent and Hysteria service output were left on systemd's default journal path. Fix: write Agent logs to `/var/log/mxinhy-agent/agent.log`, configure a Hysteria service drop-in that appends stdout/stderr to `/var/log/mxinhy-agent/hysteria.log`, and read node logs from that file.
- Symptom: Agent log lines lack a consistent timestamp and severity prefix. Cause: Agent diagnostics were written directly to stderr with ad-hoc text. Fix: route Agent diagnostics through a small logger that emits `[YYYY-MM-DD HH:mm:ss][level] message` lines before writing to `/var/log/mxinhy-agent/agent.log`.
- Symptom: preparing the repository for public release requires removing internal development history while keeping the current release snapshot. Cause: normal branch commits keep prior objects and tags reachable. Fix: create an orphan release root commit from the current tracked files, recreate only the intended release tag, force-update the protected branch through a temporary force-push window, prune local reflogs/objects, and remove embedded credentials from the remote URL.
- Symptom: public mirrors can expose deployment internals through CI configuration files. Cause: repository-level CI files are included in the release snapshot by default. Fix: keep provider CI configuration out of the public snapshot unless it is explicitly intended for users, and verify `.gitlab-ci.yml` plus `.github/` before publishing.
- Symptom: public mirror repositories can accidentally expose deployment internals or secrets once GitLab CI is committed for automation. Cause: deployment hostnames, paths, host fingerprints, or credentials are often written directly into `.gitlab-ci.yml` and deployment docs for convenience. Fix: keep GitLab CI fully generic, inject all sensitive deployment data through masked/protected CI variables, and use only placeholder domains/paths in tracked docs because the repository is mirrored publicly.
- Symptom: a public repository can still auto-deploy on other collaborators' pushes if CI only checks branch or tag names. Cause: deployment rules scoped only to `main` and release tags are too broad for public or mirrored repositories. Fix: gate deployment and release jobs on both protected refs and an explicit GitLab trigger identity such as `DEPLOY_TRIGGER_LOGIN`, so only the designated maintainer account can trigger production automation.
- Symptom: public release snapshots can accumulate tracked debug bundles, local release archives, and runtime state files even though they are already ignored. Cause: one-off troubleshooting artifacts and packaged release directories were committed before cleanup, and runtime files like `storage/security/login_attempts.json` were not consistently excluded. Fix: keep `release/`, `.debug-tools/`, `debug-*.md`, and runtime state under `storage/` out of version control, and remove historical tracked copies when preparing the repository.
- Symptom: the repository accumulates multiple explanatory `docs/*.md` files that duplicate information already present in the root README and increase maintenance overhead. Cause: implementation tasks kept generating separate handoff, flow, backend, and deployment documents even though the project is not intended for beginner-facing documentation. Fix: keep project documentation minimal by centering essential usage and architecture notes in `README.md`, and only retain `README.md`, `AGENTS.md`, and `AI_PROJECT.md` unless a truly mandatory standalone document is required.
- Symptom: README keeps growing into a long explainer with architecture walkthroughs, role descriptions, and step-by-step prose that duplicates code intent. Cause: task-driven updates append new sections without re-compressing old content. Fix: keep README as a concise operator/developer entry file focused on requirements, deployment, development, CI, and core constraints; move nonessential narration out instead of expanding it.
- Symptom: maintainer-specific identifiers can leak into tracked CI files when deployment rules are hardcoded for a single account. Cause: using a literal GitLab username inside `.gitlab-ci.yml` is a quick workaround but it embeds personal repository metadata into a public codebase. Fix: keep trigger identity configurable through a CI variable such as `DEPLOY_TRIGGER_LOGIN`, and avoid hardcoding personal usernames in tracked project files.
- Symptom: GitLab CI frontend build fails with `vue-tsc: not found` even though `package.json` declares the build tooling correctly. Cause: setting `NODE_ENV=production` globally makes `npm ci` skip `devDependencies`, but the frontend build still needs tools like `vue-tsc` and `vite`. Fix: do not force global `NODE_ENV=production` in build pipelines that run `npm ci`, or explicitly install dev dependencies for the build job.
- Symptom: GitLab CI deploy job fails with `scp: stat local "22": No such file or directory` right after SSH preparation succeeds. Cause: the deploy script reuses SSH options for `scp`, but `scp` uses uppercase `-P` for port while lowercase `-p` means preserve file attributes, so the port value is misread as a local path. Fix: keep separate option arrays for `ssh` and `scp`, using `-p` only for `ssh` and `-P` for `scp`.
- Symptom: the user page can only expose a single-node `hy2://` link, so clients cannot use one subscription entry to pull multiple nodes for the same account. Cause: connection links were assembled in the frontend from the currently selected node instead of being generated by a backend subscription endpoint. Fix: move subscription generation to the backend, expose a signed subscription URL, and aggregate subscribable entries across nodes by the same username.
- Symptom: replacing the public PHP panel entrypoint with a new Go service too early would break most existing frontend APIs before the migration is complete. Cause: the current frontend depends on a large `/api/*` surface, while the Go rewrite must be rolled out incrementally. Fix: let the Go panel take over the public entry and static assets first, and proxy unported `/api/*` requests to a separately supervised legacy PHP HTTP service until the remaining endpoints are migrated.
- Symptom: keeping legacy PHP fallback in the deployment plan makes a clean greenfield Go deployment harder to reason about and leaves unnecessary runtime dependencies in a brand-new environment. Cause: migration-oriented rollout logic mixes temporary compatibility concerns into the target architecture. Fix: for brand-new deployments, drop legacy PHP services entirely, keep Caddy/Nginx + Go panel + MySQL as the only panel runtime stack, and treat old PHP code only as migration reference until all APIs are rewritten.
- Symptom: after migrating only a subset of business APIs to Go, the dashboard can still fail to bootstrap even though login and admin management already work. Cause: the frontend startup flow also depends on supporting read-only endpoints such as `system-health`, `panel/state`, subscription info, node pagination, and service status probes. Fix: when replacing the PHP panel incrementally, migrate the full frontend bootstrap dependency chain together, and provide lightweight read-only fallbacks for status/metrics endpoints before tackling remote execution flows.
- Symptom: even after core CRUD pages are migrated, the frontend can still break because service-layer APIs outside the main dashboard remain unported. Cause: the effective contract is defined by `web/src/services/hysteriaService.ts`, which also includes `nodes/install`, `jobs/{id}`, `notification-settings`, `audit-logs`, public Hysteria auth, and Agent callback/download endpoints. Fix: during Go migration, audit the frontend service module and public machine-to-machine routes together, not just the visible admin pages, and treat them as one compatibility surface.
- Symptom: a Go endpoint can appear migrated but still return broken audit data or panic on query scans after wiring it into the frontend. Cause: the implementation guessed legacy column names like `admin_user_id` and `meta_json`, while the real schema and frontend contract still use `audit_logs.admin_id`, `details_json`, plus joined `admin_username` and `admin_display_name`. Fix: before declaring a PHP route fully replaced, verify the exact database columns against `database/schema.sql` and the exact response fields against `web/src/types.ts`, especially for audit/history style endpoints.
- Symptom: the repository can look “almost fully Go” while `app/*.php` still cannot be deleted cleanly. Cause: after the Web entrypoint is migrated, a leftover PHP CLI such as `bin/run_job.php` can still transitively require `MigrationService`, `HysteriaService`, `RemoteExecutor`, `AgentService`, and related legacy classes. Fix: when removing PHP from this project, audit CLI/cron entrypoints together with HTTP routes, add any missing Go replacements such as `run-job` and automatic database migrations first, then soft-delete the entire remaining PHP chain in one batch.
- Symptom: a login-page appearance setting such as background image seems “saved” for one browser but disappears after clearing cache, changing devices, or switching browsers. Cause: the value was stored only in frontend `localStorage`, so it was never part of the server-side system configuration. Fix: treat shared appearance settings as real system settings, persist them on the backend (database or config store), expose them through both public bootstrap settings and privileged admin settings APIs, and reserve browser storage only for truly local, per-device preferences.
- Symptom: mutable panel settings like branding, login protection, or SMTP can drift between code defaults, env files, and runtime behavior, making the real source of truth unclear. Cause: these operator-managed settings were treated like startup configuration and written back into `panel.env`. Fix: for greenfield deployments, keep only bootstrap/runtime essentials such as bind address, DB connection, and secrets in env, and persist all operator-managed settings in `system_settings` so admin changes update the database only.
- Symptom: repository cleanup keeps finding stale local artifacts such as unused editor settings, root-level test binaries, or dead TypeScript include entries after tooling changes. Cause: local build outputs and old config references stay tracked even after the runtime path changes. Fix: treat editor metadata and ad-hoc binaries as local-only, keep them ignored, and remove dead config references as soon as the related file or tool is retired.
- Symptom: the repository structure becomes hard to reason about when frontend source, deploy templates, executable entrypoints, and downloadable binaries all live under a generic `resources/` tree. Cause: one directory was carrying multiple unrelated semantics at the same time. Fix: standardize the top-level layout so `web/` holds frontend source, `cmd/` holds Go entrypoints, `deploy/systemd/` holds service templates, `artifacts/agent/` holds distributable Agent binaries, and `public/` remains only the built web output.
- Symptom: CI deploy packages can contain stale or missing Agent installers after the repository layout is standardized, even though the panel binary is rebuilt correctly. Cause: the pipeline only rebuilt `build/panel/mxinhy-panel` and relied on tracked `artifacts/agent/*` files instead of generating fresh Agent binaries during the build job. Fix: make CI always rebuild both Linux `amd64` and `arm64` Agent binaries into `artifacts/agent/` before packaging the release snapshot.
- Symptom: deployment releases can unnecessarily expose source trees or duplicate runtime assets even though the operator only needs a minimal runnable package. Cause: CI packaged a broad repository snapshot and kept external `public/` assets even after tag builds gained embedded frontend support. Fix: assemble a dedicated minimal `release-root`; keep `public/` only for non-tag compatibility packages, and let tag release packages ship the embedded panel binary plus the remaining runtime essentials only.
- Symptom: SSH password authentication for node install/restart fails in the Go panel even though the same host, user, and password work with manual `ssh`. Cause: node records are stored with encrypted sensitive fields, but the SSH execution path fetched `server_nodes` rows and passed `ssh_password` directly to the Go SSH client without decrypting it first. Fix: decrypt node secrets immediately after loading a stored node record, before any SSH, Agent bootstrap, or service-action path uses the credentials.
- Symptom: creating a new Agent-managed node fails immediately with `节点 Agent 尚未部署，请先执行安装流程`, so the node record is rolled back and cannot be added. Cause: the install pipeline restarts Hysteria through the generic `serviceAction()` path before Agent bootstrap, but `serviceAction()` switches Agent-mode nodes to `runAgentTask()`, which requires an already-registered Agent. Fix: during first install, always restart Hysteria over SSH before bootstrapping the Agent; only use Agent task-based service control after the bootstrap step has completed.
- Symptom: saving runtime settings such as `public_api_base_url` in the admin backend updates the database but some request paths still behave as if the old value were active. Cause: `App` kept its own copied `Config` value while long-lived services such as `hysteria`, `auth`, `agents`, `systemHealth`, and `loginProtection` were constructed with a pointer to a different `Config` instance created during startup. Fix: keep the web app and all long-lived services on the same shared `*Config` object, and when setup reinitializes configuration replace the pointed value instead of swapping only one copy.
- Symptom: manually replacing the panel binary from a macOS workstation can make `systemd` fail with `status=203/EXEC` on the Linux server. Cause: a local default `go build` on macOS produces a Mach-O binary for Darwin, which cannot execute on the remote Linux host. Fix: any manual hot replacement for the server must use an explicit Linux target such as `GOOS=linux GOARCH=amd64 CGO_ENABLED=0 go build`, and verify the remote file type before restarting the service.
- Symptom: creating or deploying an Agent-managed node can still fail with `Agent 无法访问面板接口` even though `public_api_base_url` is already configured. Cause: request handlers passed the panel origin like `https://panel.example.com` into Agent bootstrap and Hysteria auth/config builders, but those paths expect an API base URL that already includes `/api`, so remote calls were sent to `/agent/register` instead of `/api/agent/register`. Fix: keep a dedicated API-base resolver for remote node flows, prefer the configured `public_api_base_url`, and only fall back to `request origin + /api` when needed.
- Symptom: Agent bootstrap can finish and write `mxinhy-agent.service`, but the service immediately exits with `status=209/STDOUT` on some Debian hosts. Cause: the unit template relied on `StandardOutput=append:...` / `StandardError=append:...`, which failed in the target systemd environment during stdio setup. Fix: keep the service template on a portable shell redirection form such as `/bin/sh -c 'exec ... >>/var/log/mxinhy-agent/agent.log 2>&1'` instead of depending on `append:` output directives.
- Symptom: subscription QR codes or copied subscription links can still point at `127.0.0.1` or malformed `/api/subscription/...` paths after `public_api_base_url` is configured. Cause: the codebase mixed two different concepts, using the same value as both the public site root and the public API root, while subscription routes actually live under `/subscription/` and Agent/Hysteria callbacks live under `/api/...`. Fix: normalize and resolve public site base URLs and public API base URLs separately, store `public_api_base_url` in canonical `/api` form, derive site-root links by trimming `/api`, and make mock/frontend helpers match the real `/subscription/` route.
- Symptom: restarting `mxinhy-panel.service` can suddenly fail with `status=203/EXEC` after a release switch even when the binary built locally is valid. Cause: the active systemd unit may already point at a newer release directory whose `build/panel/mxinhy-panel` file was never packaged or copied, so restarting the service drops onto a missing executable instead of the previously running release. Fix: before any hot replacement or restart, verify the exact release path referenced by `ExecStart`, ensure that path contains a Linux panel binary, and repair the active release directory rather than assuming the old `current` target is still the one in use.
- Symptom: a newly created Hysteria node appears online and its server process is running, but every client connection fails immediately and local `hysteria ping` reports `authentication error, HTTP status code: 404`. Cause: the Go panel only registered `GET /api/hysteria/auth/`, while the Hysteria server performs HTTP auth with `POST`, so requests fell through to the Go panel fallback and returned the generic “not migrated” response. Fix: register both `GET` and `POST` for `/api/hysteria/auth/`, then verify with a real local Hysteria client plus a proxy data-path test instead of relying only on service status.
- Symptom: agent-managed nodes suffer slow first connections on high-latency links and can keep serving stale local access rules when the control plane becomes unreachable. Cause: node auth still depends on per-connection panel HTTP callbacks, while the Agent previously exited on register failure and had no local heartbeat guard for Hysteria service access. Fix: keep agent-managed nodes on local `userpass` auth synced asynchronously from the panel, make the Agent survive register/heartbeat failures, and stop Hysteria locally after the missed-heartbeat grace window until panel heartbeats recover.
- Symptom: agent-managed nodes still disconnect clients whenever user state changes, even after auth was moved off the panel callback path. Cause: syncing users through static `userpass` config still requires rewriting `config.yaml` and restarting Hysteria for password, status, or quota changes. Fix: keep Hysteria on a localhost Agent auth callback, sync user state separately into the Agent's local auth store, and let heartbeat loss reject new connections in the callback layer instead of stopping the Hysteria service.
- Symptom: GitLab CI deployment can finish successfully but the live site still serves an old panel/frontend, and `current` may even point to a missing release directory. Cause: release cleanup sorted release directories by commit SHA name instead of deployment time, so a newly deployed SHA like `c...` could be treated as “old” and deleted immediately after switching the symlink. Fix: prune releases by modification time, not directory name, and never delete the directory currently referenced by `current`.
- Symptom: after a release switch, the panel can return `404 page not found` for `/` even though the new release contains `public/index.html`. Cause: `PANEL_STATIC_DIR` was resolved to an absolute release path at runtime and then written back into `panel.env`, so later deployments kept pointing static file serving at a deleted old release directory. Fix: keep `PANEL_STATIC_DIR` persisted as a stable relative setting like `public`, and if a saved absolute path points into an old `releases/.../public`, automatically fall back to the current working directory's `public`.
- Symptom: the admin-side version upgrade path can break both fixed-install deployments and `releases/current/shared` deployments even when tag detection works. Cause: the old upgrader downloaded the repository source archive instead of the runtime deploy package, so it could not faithfully reproduce built binaries, frontend assets, or release-layout switching. Fix: publish the CI-built runtime deploy package to a stable tag-based download URL, and make the upgrader consume that package while detecting whether the target is a fixed root or a rolling `releases/current/shared` installation.
- Symptom: the latest GitLab tag pipeline can fail in `build_release` with `curl: not found` after frontend and Go binaries already build successfully. Cause: the Alpine build job installs `nodejs` and `npm` but the tag-only runtime package upload step also depends on `curl`, which is not present in the base `golang:alpine` image. Fix: install `curl` in `build_release` `before_script` whenever the job may upload the deploy package to the GitLab Package Registry.
- Symptom: releasing through `build.sh` updates Git tags and `package.json`, but the sidebar can still show `vdev` or a stale frontend version after deployment. Cause: the UI reads `import.meta.env.VITE_APP_VERSION`, while the build pipeline previously never injected that variable from `package.json`. Fix: make the shared frontend build entrypoint load `package.json.version` and export it as `VITE_APP_VERSION` before running `vue-tsc`, `vite build`, and obfuscation, so local builds and CI releases stay consistent.
- Symptom: sometimes a maintainer needs to publish a protected version tag for upgrade-package validation without triggering the production deployment job. Cause: tag pipelines normally share the same `deploy_production` rules as real releases, so pushing a test tag would also deploy online. Fix: gate `deploy_production` on an optional CI variable such as `SKIP_PRODUCTION_DEPLOY`, then push the validation tag with that variable set to `1` so build and release still run while deployment is skipped.
- Symptom: GitLab Release pages can look wrong for operators by showing only default source archives or by exposing an inconsistent runtime package name. Cause: the tag pipeline uploaded the real upgrade package separately to the GitLab Package Registry, while the Release job originally omitted an asset link and the archive filename was hardcoded as `deploy-package.tar.gz` across multiple release paths. Fix: expose the runtime package on the Release page with the operator-facing label `hysteria2-plan` and standardize the actual archive filename to `hysteria2-plan.tar.gz` across CI packaging, Package Registry upload, deploy script defaults, and panel upgrade downloads.
- Symptom: a tag release panel binary can still return site-root 404s if the runtime working directory or external `public/` path does not line up with `PANEL_STATIC_DIR`. Cause: the normal panel binary reads frontend assets from disk, so release environments remain sensitive to working-directory and path mismatches. Fix: build tag-release panel binaries with a dedicated `release_embed` build tag that embeds the current `public/` assets into `mxinhy-panel`, and keep the server on a disk-first, embedded-fallback static serving path.
- Symptom: first boot still feels coupled to a pre-created `panel.env`, even though the panel can already start with safe defaults and only needs durable secrets/DB settings after setup. Cause: startup docs and flags treated `-config` as mandatory, while the code path already tolerated a missing env file by using defaults. Fix: default the binary to `config/panel.env`, allow startup without that file, write the config automatically after first setup completes, and expose a separate CLI bind override such as `-port` for temporary listen changes without editing the persisted config.
- Symptom: setup can fail with `open database/schema.sql: no such file or directory` or the panel can miss `public/` when the service starts from `build/panel/` or another working directory. Cause: relative runtime paths were previously resolved from the current shell/service cwd instead of from the deployed project layout. Fix: resolve the default config path, project root, schema path, SSH helper root, and relative static dir from the persisted config path first and otherwise from the real `mxinhy-panel` executable path, only falling back to cwd as a last resort.
- Symptom: login can break after release when CI injects a frontend `VITE_LOGIN_AES_SEED` that does not match the runtime `PANEL_LOGIN_AES_SEED` later written into `panel.env` during first setup. Cause: the browser-side password wrapper previously depended on a build-time long-lived seed while the backend decrypt path used the runtime config seed. Fix: stop injecting a frontend long-term seed, have `/api/auth/login-challenge` return a per-request derived seed based on the runtime config and current nonce, and let the frontend encrypt the password with that one-time value only.
- Symptom: Agent bootstrap in a rolling `releases/current/shared` deployment can fail with `Agent systemd 模板不存在` even though the release package contains `deploy/systemd/mxinhy-agent.service`. Cause: deriving the project root from `shared/config/panel.env` points runtime resource lookups at `shared/`, while release-owned assets actually live under the current release directory. Fix: when the persisted config path is `shared/config/panel.env`, resolve runtime resources from the current release root first, falling back to the `current` symlink under the deploy root instead of treating `shared/` as the project root.
- Symptom: changing `PANEL_MOCK_PANEL` in `panel.env` does not reliably switch the mock experience and cannot cover richer mock scenarios like configurable node counts or status mixes. Cause: the old mock toggle was file-based, only surfaced through `/api/mock-mode`, and the frontend cached the flag locally while generating mock data from hardcoded arrays. Fix: move mock panel controls into database-backed `system_settings`, expose them through the system settings API and `/api/mock-mode`, and regenerate frontend mock nodes/users/statuses from the saved settings while syncing the client cache immediately after saves.
- Symptom: first-time node install fails on minimal Linux hosts because the default Hysteria install script needs `curl`, Agent bootstrap needs a downloader, and self-signed mode also needs `openssl`, but fresh systems may not include them. Cause: the install flow assumed those tools already existed on the remote host and only surfaced a late failure during install or Agent bootstrap. Fix: run a dependency preflight before first install, auto-install missing `curl` (and `openssl` for `self_signed`) through common Linux package managers, and fail early with a clear message when the SSH account lacks usable root/sudo privilege.
- Symptom: deleting a node makes users disappear from the panel. Cause: `hysteria_users.node_id` has `ON DELETE CASCADE`, so deleting `server_nodes` directly removes node-scoped user records. Fix: delete nodes through a transaction that migrates users to a compatible remaining node first, and block deletion when no conflict-free destination exists.
- Symptom: a subscription link imports in v2rayNG but OpenClash cannot recognize it. Cause: OpenClash expects Clash/Mihomo YAML, while v2rayNG accepts raw `hy2://` URI lines. Fix: keep raw URI output as the default, but detect OpenClash/Clash/Mihomo user agents or `format=clash` query parameters and return a Clash YAML subscription with all eligible Hysteria2 nodes.
