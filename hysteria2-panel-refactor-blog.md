# 利用 Hysteria2 面板项目搭建节点

最近把自己写的这个 Hysteria2 面板项目整理得差不多了，顺手写一篇搭建记录，免得后面又忘了。

这个项目不是单纯的“生成配置文件”，而是完整的面板 + 节点管理方案：

- 面板负责用户、节点、订阅、系统设置
- 节点负责运行 Hysteria2
- Agent 负责接管节点的状态上报、日志读取、配置同步和服务控制

如果只是想快速搭一个可管理的节点，这套东西用起来还是挺顺手的。

## 先说一下整体流程

利用这个项目搭建节点，大概就是下面这几步：

1. 先把面板部署起来
2. 首次进入面板完成初始化
3. 在系统设置里配置面板公网 API 地址
4. 添加节点 SSH 信息和 Hysteria 参数
5. 让面板通过 SSH 自动安装 Hysteria2
6. 如果选择 Agent 模式，面板会继续把 Agent 一并引导上去
7. 后续节点的配置下发、状态查看、日志读取、服务启停，都直接在后台处理

如果你问我推荐哪种方式，我还是更建议直接用 Agent 模式。

## 准备工作

开始之前，建议先准备好下面这些东西：

- 一台部署面板的服务器
- 一台作为节点的服务器
- MySQL 或 MariaDB 数据库
- 一个能正常解析到面板的域名
- 一个给节点使用的域名
- 节点服务器可正常使用 SSH 登录
- 节点服务器安装了 `systemd`
- 节点服务器至少有 `curl` 或 `wget` 其中一个

如果节点机器连 `curl` 和 `wget` 都没有，后面 Agent 是拉不下来的，这个需要提前注意一下。

## 面板部署

### 运行环境

面板这边的运行环境要求并不复杂：

- Go `>= 1.23`
- MySQL `>= 5.7` 或 MariaDB `>= 10.2`
- Node.js `>= 18`
- Nginx 或 Caddy 反向代理

项目本身是 `Go Panel + MySQL + Vue 3 + TypeScript`，前端打包后由 Go 直接提供页面，部署起来不算折腾。

### 基础部署思路

先把项目放到服务器上，然后准备面板配置文件。  
项目里已经有一个示例文件：

```bash
config/panel.env.example
```

正常情况下复制一份作为自己的运行配置即可，比如：

```bash
cp config/panel.env.example config/panel.env
```

然后把数据库连接、面板监听地址这类基础信息配好，再启动面板。

如果是自己开发环境测试，也可以直接这样跑：

```bash
go run ./cmd/mxinhy-panel -config config/panel.env.example
```

生产环境的话还是建议自己编译成二进制再挂到 `systemd` 或其他守护方式里。

## 首次初始化

面板第一次打开时会进入首次安装流程。

这里需要填写的核心内容主要有：

- MySQL 主机
- MySQL 端口
- 数据库名
- 数据库用户名和密码
- 面板公网 API 地址
- 管理员账号
- 管理员密码

这里有一个比较关键的点：

### `public_api_base_url` 一定要填对

这个地址不是你自己浏览器访问后台的随便一个地址，而是“节点服务器可以访问到的面板 API 地址”。

比如面板域名是：

```text
https://panel.example.com
```

那这里建议填成：

```text
https://panel.example.com/api
```

因为节点在安装 Agent、注册 Agent、心跳上报、拉取任务的时候，都是要访问这个地址的。

如果这里没配好，最常见的现象就是：

- 节点能 SSH 连上
- Hysteria 能装上
- 但 Agent 部署失败
- 后台提示 Agent 无法访问面板接口

这个坑我自己已经踩过了，所以这里单独提一下。

## 添加节点

面板初始化完成后，就可以开始加节点了。

新建节点时页面上主要要填这些内容：

- 节点地址或域名
- SSH 端口
- SSH 用户名
- SSH 认证方式
- 监听端口
- 证书模式
- 运维模式
- Hysteria 服务名
- 配置文件路径

其中运维模式有两个：

- `Agent`
- `SSH`

我这边默认更推荐 `Agent`，因为后续很多操作会更方便。

### SSH 认证

SSH 认证可以用密码，也可以用私钥。

如果你用私钥方式，建议直接在面板里上传 SSH 私钥，不要自己手填服务器上的路径。  
这样会更安全一点，后面节点安装时也更省心。

### 证书模式

目前面板里主要支持两种：

- `self_signed`
- `acme`

如果只是自己先搭起来跑通流程，用自签证书会简单一点。  
如果是正式使用并且域名、80/443 条件都满足，可以直接走 ACME。

## 点击安装后，面板到底做了什么

这个项目的节点安装并不是只跑一个命令结束，而是按顺序做了几件事：

### 1. 先通过 SSH 执行安装脚本

面板会先连到你的节点服务器，执行节点配置中的安装脚本，把 Hysteria2 运行环境装好。

### 2. 处理 TLS 资源

如果你选的是自签证书模式，面板会自动在节点上生成对应的证书和私钥文件。

### 3. 下发 Hysteria 配置

安装完成后，面板会把当前节点配置和用户配置整理成 Hysteria 配置文件，再通过 SSH 写入节点指定的配置路径。

### 4. 重启 Hysteria 服务

配置落地后，面板会直接执行对应服务的重启动作，让节点配置立即生效。

### 5. 如果是 Agent 模式，再继续引导 Agent

这一步是整个项目里我比较喜欢的地方。

当节点运维模式是 `Agent` 时，面板会继续做下面这些事：

- 检测节点 CPU 架构
- 自动选择对应的 Agent 二进制
- 通过面板提供的签名下载地址拉取 Agent
- 写入 Agent 配置文件
- 写入 `systemd` 服务文件
- 启动并注册 Agent

也就是说，第一次节点安装完成后，后面的很多事情就不用再完全依赖 SSH 了。

## Agent 模式有什么用

很多人看到 Agent 可能会问一句：我都能 SSH 了，为什么还要多一个 Agent？

因为后面真的会方便很多。

Agent 接管之后，可以做这些事：

- 定时向面板上报节点状态
- 拉取面板下发的配置任务
- 执行服务启停和重启
- 返回节点日志
- 同步本地流量统计
- 接收本地授权数据

尤其是现在这个项目已经把鉴权链路改成了“本地 Agent 鉴权”，节点在高延迟环境下会舒服很多。

简单说就是：

- 面板负责控制面
- Agent 负责节点本地执行
- Hysteria 负责真正提供服务

这种分层比单纯靠 SSH 一把梭要稳定一些。

## 后续怎么管理节点

节点装好之后，后面常见操作基本都可以直接在面板完成：

- 重启服务
- 停止服务
- 查看运行状态
- 下发新配置
- 查看节点日志
- 同步流量统计
- 升级 Agent

如果用户有变更，比如：

- 新增用户
- 用户停用
- 流量变化
- 到期时间变化

面板后续也可以继续把更新同步给节点，不需要你每次都手动上去改配置。

## 搭建时几个容易踩的坑

### 1. 面板公网 API 地址没配好

这是最常见的问题。

如果 `public_api_base_url` 填错，或者节点机器根本访问不到面板地址，那么 Agent 部署大概率会失败。

### 2. 节点没装 `curl` 或 `wget`

因为 Agent 首次引导时需要在节点机器上下载对应架构的二进制，所以至少得有一个下载工具。

### 3. SSH 能连，但 sudo 或 systemd 不正常

有些机器虽然能 SSH 上去，但执行安装脚本、写服务文件、重启服务的时候会翻车。  
尤其是精简系统或者权限不完整的环境，要提前确认一下。

### 4. 域名和证书条件没准备好

如果你走 ACME，但是域名解析、端口放行或者证书条件不满足，那证书签发阶段就容易出问题。  
图省事的话，先自签把流程跑通，再慢慢换正式证书也行。

## 我自己的建议

如果只是想先把节点跑起来，我建议这样用：

1. 面板先部署好
2. 首次初始化时把 `public_api_base_url` 配正确
3. 新建节点时直接选 `Agent`
4. 证书模式先选自签
5. 确保节点有 `curl` 或 `wget`
6. 先把整条安装链路跑通

等节点能正常在线、能下发配置、能看日志以后，再去慢慢调整：

- 域名
- ACME
- 带宽参数
- 混淆参数
- 其他高级配置

这样会省掉很多排错时间。

这个项目现在用来搭 Hysteria2 节点，已经不是那种“半自动写个配置文件”的阶段了，而是基本可以做到：

- 面板统一管理
- 节点自动安装
- Agent 自动接管
- 后续配置持续同步
